# 09. Workflow templates and editor

## Цель

Добавить отдельную страницу редактора, которая использует поддерживаемые workflow templates, проверяет доступность зависимостей и позволяет запускать генерацию через подключённый ComfyUI.

Извлечение workflow из metadata существующего изображения остаётся полезной возможностью, но не должно быть единственным или основным способом генерации.

## Почему нужны шаблоны

Workflow, полученный из чужого изображения, может ссылаться на отсутствующие custom nodes, checkpoints, LoRA, VAE, ControlNet и другие ресурсы. Приложение не может гарантировать его воспроизводимость.

Поддерживаемый template предоставляет известную структуру и понятные редактируемые поля. На первом этапе нужны категории:

- простая генерация;
- генерация по sample/reference;
- видео;
- расширенная многоэтапная генерация.

Конкретный набор templates можно расширять без изменения основной страницы редактора.

## Формат template

Workflow JSON должен сопровождаться manifest, который описывает:

- ID, имя, версию и категорию;
- media type результата;
- файл workflow;
- обязательные ноды;
- необходимые типы model resources и поддерживаемые semantic slots;
- редактируемые параметры;
- привязку полей UI и resource slots к semantic role, node input или другому проверяемому binding;
- выходные ноды;
- необязательное preview и описание.

Пользовательские templates допускается импортировать через JSON или архив с manifest, workflow и дополнительными файлами.

Manifest не должен содержать закрытый поимённый список всех checkpoint и LoRA, которые пользователь может выбрать. В обычном случае он объявляет capabilities и slots, например один checkpoint подходящей ecosystem, динамическую цепочку совместимых LoRA, optional VAE или ControlNet. Конкретный локальный ресурс выбирается из общего каталога задания 07 и повторно валидируется при открытии и запуске draft.

Для стандартного однозначного графа binding может автоматически находить `CheckpointLoaderSimple`, `LoraLoader`, prompt encoders и их typed edges. Несколько LoRA могут подключаться generic graph transformation, которая строит цепочку стандартных loader nodes, вместо отдельных вариантов template для каждого количества или имени LoRA. Если workflow содержит несколько независимых model pipelines, нестандартные custom nodes или другую неоднозначность, автор template явно размечает semantic slots. Такая разметка является декларативным контрактом графа, а не хардкодом конкретных моделей в UI.

Пример направления manifest, не финальная схема:

```yaml
resource_slots:
  checkpoint:
    accepts: [checkpoint]
    binding: base_model
  loras:
    accepts: [lora, locon, dora]
    multiple: true
    binding: model_clip_chain
```

Окончательную схему bindings нужно выбрать после проверки реальных API workflows ComfyUI, стандартных loader nodes, custom-node variants, model-only и model-plus-CLIP LoRA, нескольких pipelines, видео workflows и round-trip сохранения metadata. Пример выше фиксирует принцип, но не предопределяет реализацию.

## Проверка зависимостей

Перед открытием или запуском template приложение должно сравнить его требования с подключённым ComfyUI:

- доступны ли нужные node types;
- присутствуют ли выбранные model resources и можно ли разрешить каждый semantic slot;
- совместимы ли checkpoint, дополнительные resources и architecture workflow;
- совместим ли template с текущим runtime.

Отсутствующие зависимости показываются списком. Для custom nodes позднее можно добавить интеграцию с ComfyUI Manager и установку только после явного подтверждения пользователя.

Модели приложение не обязано искать и скачивать автоматически. Достаточно показать стабильную identity или точное имя, тип ресурса, ожидаемую ecosystem и место использования в workflow. Внешняя metadata может помогать поиску, но не является обязательной для локального запуска.

## Редактор

Редактор должен быть отдельной HTML-страницей и строить форму на основе manifest. Пользователь видит только значимые параметры, а не весь сырой граф ComfyUI.

Минимально нужны:

- positive и negative prompt;
- model/checkpoint;
- seed;
- размеры;
- steps, CFG, sampler и scheduler там, где они применимы;
- reference/sample inputs;
- параметры многоэтапного workflow;
- preview итогового workflow перед запуском.

Допускается отдельный расширенный режим для просмотра или ручной правки JSON, но он не должен быть основным интерфейсом.

В качестве одного из визуальных ориентиров допускается использовать [экспериментальный двухпанельный референс](../../image.png): controls слева, результаты справа, компактные карточки checkpoint/LoRA и actions выбранного результата. Это только пример реализации. Финальный UI нужно выбрать с учётом всех типов workflows, resource compatibility, SceneSpec и Remix, очереди, ошибок, accessibility и responsive layout.

Основной интерфейс должен оставаться лёгким и контекстным. Manifest и resolved capabilities определяют видимые controls; advanced параметры сворачиваются, а отсутствующие в выбранном workflow поля не рендерятся. Raw graph, node IDs и технические bindings не должны занимать основной экран.

## Извлечённый workflow и Remix

Workflow из metadata можно открыть для анализа и попытаться проверить. Если зависимости отсутствуют, пользователь получает отчёт и может перейти к поддерживаемому template.

Команда Remix должна:

1. взять prompt или AI-реконструкцию выбранного asset;
2. предложить template;
3. создать заполненный черновик;
4. открыть редактор;
5. дождаться ручного запуска пользователя.

## Выполнение

После запуска редактор должен показывать очередь, прогресс, ошибки и результат. Созданный asset автоматически возвращается в библиотеку и при необходимости связывается с исходным изображением как remix.

## Критерии готовности

- Редактор существует как отдельная страница.
- Templates отделены от UI и описываются manifest.
- Импорт template не требует изменения исходного кода для каждого нового workflow.
- Manifest объявляет semantic resource slots, а не фиксированный поимённый набор checkpoint и LoRA.
- Стандартные checkpoint и LoRA bindings обрабатываются generic resolver/graph transformation без отдельного кода формы для каждого template.
- Редактор получает model resources из динамического каталога и повторно проверяет compatibility перед запуском.
- UI остаётся контекстным и не перегружает основной экран advanced или неприменимыми параметрами.
- Добавленный визуальный референс используется как ориентир, но не считается финальной спецификацией.
- До запуска проверяются ноды и модели.
- Отсутствующие custom nodes и модели показываются раздельно.
- Remix создаёт черновик и не запускает генерацию автоматически.
- Прогресс и ошибки ComfyUI отображаются в редакторе.
- Результат автоматически появляется в локальной библиотеке.
