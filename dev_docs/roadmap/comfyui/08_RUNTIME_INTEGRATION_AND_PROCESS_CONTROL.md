# 08. Runtime integration and process control

## Цель

Подключить стандартную локальную установку ComfyUI, автоматически подготовить команду запуска для текущей ОС и управлять запущенным процессом из интерфейса ComfyUI Meta Viewer.

Windows, Linux и macOS должны поддерживаться на равных условиях.

## Выбор директории

Пользователь указывает путь к установке. Приложение должно принять два распространённых варианта:

- выбранная директория сама содержит `main.py`;
- выбранная директория содержит вложенную `ComfyUI/main.py`.

Например, оба пути должны распознаваться корректно:

```text
F:\ComfyUI_windows_portable
F:\ComfyUI_windows_portable\ComfyUI
```

Определение выполняется по реальной структуре и наличию entry point, а не только по имени папки.

Если обнаружено несколько нестандартно вложенных версий или структура не соответствует поддерживаемым вариантам, приложение показывает ошибку. Поддержка любой пользовательской организации файлов не является обязанностью проекта.

## Поиск интерпретатора

Для стандартной Windows Portable установки следует искать встроенный Python рядом с директорией ComfyUI, например `python_embeded/python.exe`.

Для обычных установок проверяются распространённые виртуальные окружения:

- Windows: `.venv/Scripts/python.exe`, `venv/Scripts/python.exe`;
- Linux/macOS: `.venv/bin/python`, `venv/bin/python`.

При необходимости позднее можно добавить экспертный режим с ручным указанием исполняемого файла и аргументов. Основной сценарий должен оставаться автоматическим.

## Скрипты запуска

После успешного обнаружения приложение создаёт прозрачный пользовательский скрипт:

- `.bat` для Windows;
- `.sh` для Linux и macOS.

Скрипт содержит абсолютные безопасно экранированные пути, рабочую директорию и необходимые аргументы запуска. Он нужен как проверяемый и доступный запасной способ запуска.

Для управления из UI backend должен по возможности запускать найденный Python напрямую через `subprocess`, а не через shell. Это позволяет надёжно получить PID, читать stdout/stderr и завершать только принадлежащий приложению процесс.

## Managed и external режимы

### Managed

ComfyUI запущен из Meta Viewer. Приложение может:

- показывать stdout и stderr;
- хранить PID;
- сообщать об аварийном завершении;
- останавливать и перезапускать процесс.

### External

ComfyUI уже запущен отдельно. Приложение подключается к API, показывает состояние и отправляет workflow, но не должно завершать чужой процесс.

## Статус и здоровье

Интерфейс должен различать как минимум:

- `stopped`;
- `starting`;
- `ready`;
- `busy`;
- `stopping`;
- `error`;
- `external`.

После запуска нужно ждать готовности API, а не считать процесс рабочим только потому, что он существует. Логи, health status, очередь и ошибки должны отображаться без необходимости открывать отдельный терминал.

Команды `Прервать генерацию` и `Остановить ComfyUI` являются разными действиями.

## Критерии готовности

- Стандартная установка распознаётся при выборе её корня или вложенной папки `ComfyUI`.
- Windows Portable, обычные Windows-установки и POSIX venv обрабатываются раздельно.
- Для текущей ОС генерируется рабочий `.bat` или `.sh`.
- Managed процесс запускается, логируется и корректно останавливается из UI.
- External процесс можно использовать без попытки присвоить его PID или завершить.
- Ошибка пути, Python, запуска или API отображается отдельно и понятно.
- Нестандартная структура не запускается случайной или угаданной командой.
