# 01. Database and index reset

## Цель

Сделать локальный индекс предсказуемым в разработке и дать пользователю простой способ полностью пересоздать его, если обычный reset не помогает.

На текущем этапе база не содержит незаменимых данных, поэтому сложные миграции, автоматическое лечение повреждений и восстановление отдельных страниц SQLite не требуются.

## Требуемое поведение

Нужно разделить как минимум две операции.

### Reset index

Полностью удаляет индекс и производные данные:

- основной файл SQLite;
- связанные `-wal` и `-shm` файлы;
- кэш thumbnails;
- кэш cutouts и другие безопасно пересоздаваемые результаты.

После удаления приложение создаёт чистую схему и может повторно проиндексировать сохранённые активные источники.

### Factory reset

Дополнительно удаляет пользовательскую конфигурацию приложения. Эта операция должна быть явно отделена от обычного пересоздания индекса и требовать отдельного подтверждения.

## Хранение конфигурации

Список подключённых директорий и базовые настройки желательно хранить отдельно от пересоздаваемого индекса. Это позволит удалить повреждённую базу и заново собрать коллекцию без повторного ввода всех путей.

При появлении альбомов, избранного и пользовательских аннотаций правила reset нужно пересмотреть: интерфейс должен честно сообщать, какие данные будут потеряны.

## Ограничения

- Не требуется автоматический repair повреждённой SQLite.
- Не требуется полноценная система миграций до тех пор, пока схема активно меняется и база может быть пересоздана.
- Ошибка удаления одного из файлов должна быть показана пользователю, а не скрыта.
- Нельзя удалять исходные изображения и видео.

## Критерии готовности

- Повреждённый или заблокированный индекс можно физически удалить из интерфейса или понятной служебной команды.
- После reset база, схема и кэш создаются заново.
- Исходные пользовательские файлы не затрагиваются.
- Сохранённые источники можно повторно проиндексировать без ручного добавления каждого пути.
- Factory reset визуально и логически отделён от Reset index.
